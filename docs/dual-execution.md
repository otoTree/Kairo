# Kairo 双层执行模型

本文档定义 Kairo 的 `Dual Execution Model`。  
它回答的问题是：

如果未来的 agent 系统既需要低延迟、深度融合的模型内计算，又需要强隔离、可验证、可审计的外部程序执行，那么这两类执行能力如何在同一套系统中共存。

这份文档的直接背景来自两条正在汇合的路线：

- 一条路线把程序逻辑内化到模型参数和推理路径中，使 transformer 本身具备执行能力
- 另一条路线把程序运行放在受限、可验证、可隔离的执行域中，例如 WebAssembly 沙箱

Kairo 不应把这两条路线看成互斥替代关系，而应把它们视为两种不同层次、不同目标的执行能力，并提供统一编排。

## 1. 问题定义

未来 agent 系统面对的任务，并不都是同一种“执行”：

- 有些任务是短程、频繁、结构稳定、需要低延迟的机械计算
- 有些任务是需要权限控制、环境交互、结果审计、可重放验证的工具调用
- 有些任务则介于两者之间，需要先做内部快速计算，再做外部受控执行

如果只保留一种执行路径，会带来明显缺陷：

- 只保留模型内执行，系统会失去隔离、审计、环境控制和部署灵活性
- 只保留外部沙箱执行，系统会为大量细粒度计算支付过高的编排和往返成本

因此 Kairo 应明确支持两类执行：

1. `Parametric Execution`
2. `Sandboxed Execution`

## 2. 总体定义

### 2.1 Parametric Execution

`Parametric Execution` 指的是：  
程序逻辑以某种形式内化到模型参数、模型内部执行器或推理快路径中，并在同一次模型推理过程中被执行。

它的典型形式包括：

- 模型内部解释某种程序表示
- 模型内部执行结构化 trace
- 某段逻辑被直接编译进权重
- 模型在 forward / decode 过程中完成状态推进

### 2.2 Sandboxed Execution

`Sandboxed Execution` 指的是：  
程序逻辑在外部受限执行环境中运行，该环境拥有独立的对象边界、权限约束、可验证轨迹和可治理生命周期。

在 Kairo 的上下文中，这类执行最自然的载体就是 `WasmDomain`。

它的典型形式包括：

- 运行 WebAssembly 模块
- 带 capability 的工具调用
- 可审计的系统服务扩展
- 可重放的安全任务执行

## 3. 为什么需要双层执行

Kairo 的目标不是“让模型像计算器”，也不是“做一个会跑工具的操作系统”，而是成为一个 agent-native system。  
这意味着系统必须同时处理两类根本不同的需求。

### 3.1 低延迟内部计算

一些计算如果每次都交给外部执行环境，会产生不必要的开销：

- 需要跨边界传参
- 需要等待执行返回
- 需要上下文同步
- 需要额外 trace 注入

对于高频、机械、结构化的计算，这会让 agent 系统不必要地变慢。

### 3.2 强边界外部执行

而另一些行为不能交给模型内执行：

- 文件系统访问
- 网络访问
- 设备访问
- 持久化写入
- 受约束工具调用
- 需要外部审计和权限控制的任务

这些任务必须有外部边界，否则系统无法保障安全、治理和可靠性。

### 3.3 双层协同优于单层极端

因此更合理的系统设计是：

- 把低延迟、确定性、可压缩的计算放进 `Parametric Execution`
- 把需要隔离、审计、环境接入的逻辑放进 `Sandboxed Execution`
- 由统一编排层决定任务分流

## 4. Parametric Execution

### 4.1 定义

在 Kairo 中，`Parametric Execution` 指的是在模型本身的执行过程中完成某段程序化逻辑，而不是调用外部工具。

它可以有多种具体实现形式：

- 程序 token 化后由模型内部解释执行
- 解释器行为被编码进权重
- 某些可复用逻辑直接被编译进权重
- 推理快路径中存在专门的执行子系统

### 4.2 适合的任务

适合进入 `Parametric Execution` 的任务通常具有这些特征：

- 结构稳定
- 规则明确
- 状态空间可控
- 高频调用
- 需要低延迟
- 不涉及强副作用

典型例子包括：

- 算法性中间计算
- 结构化约束求解
- 短程程序解释
- token 级状态推进
- agent 中的局部机械推导

### 4.3 优势

- 延迟低
- 融合度高
- 不需要外部 round-trip
- 可与模型推理过程共同训练
- 可以形成 differentiable execution path

### 4.4 局限

- 不适合强副作用任务
- 难以做强隔离
- 难以做系统级权限审计
- 更新和部署复杂
- 对行为可解释性和回滚不友好

## 5. Sandboxed Execution

### 5.1 定义

在 Kairo 中，`Sandboxed Execution` 指的是在独立执行边界中运行程序对象。  
它不属于模型权重内部逻辑，而是系统外部但受治理的执行能力。

推荐的主要载体是 `WasmDomain`。

### 5.2 为什么是 Wasm

Wasm 很适合作为 Kairo 的沙箱执行对象，因为它具有：

- 指令规整
- 语义明确
- 可移植
- 确定性强
- 容易做 capability 边界
- 易于生成和验证执行 trace

这使它既适合 agent 工具运行，也适合作为受控扩展模块形态。

### 5.3 适合的任务

适合进入 `Sandboxed Execution` 的任务通常具有这些特征：

- 需要访问系统资源
- 需要权限控制
- 需要审计
- 需要结果可重放
- 需要与外部环境交互
- 需要可独立部署和升级

典型例子包括：

- 工具调用
- 文件处理
- 网络请求
- 设备操作
- 结构化数据转换
- 可验证的业务逻辑模块

### 5.4 优势

- 强隔离
- 强治理
- 强审计
- 可重放
- 易于版本管理
- 易于热更新

### 5.5 局限

- 需要跨边界调度
- 存在执行往返成本
- 与模型推理不天然可微
- 高频小任务时可能过重

## 6. 两层执行的核心区别

可以用一句话概括：

- `Parametric Execution` 是“逻辑在模型内部执行”
- `Sandboxed Execution` 是“逻辑在系统边界外部受控执行”

二者的差异可以进一步理解为：

### 6.1 位置

- Parametric：发生在模型推理循环内部
- Sandboxed：发生在模型外部的执行域中

### 6.2 状态

- Parametric：状态主要存在于权重、隐藏状态或 trace 中
- Sandboxed：状态主要存在于系统对象和沙箱实例中

### 6.3 治理

- Parametric：治理能力弱，更适合性能敏感计算
- Sandboxed：治理能力强，更适合系统级动作

### 6.4 可验证性

- Parametric：可通过 trace 观察，但不天然具备系统级审计
- Sandboxed：更容易做可重放与审计

## 7. 与 Wasm 的两种关系

Wasm 在 Kairo 中不应只扮演一种角色，而可能同时存在两种位置。

### 7.1 `WASM-as-Weights`

这表示：

- 模型内部解释或执行某种 Wasm 表示
- Wasm 语义以权重或内部执行逻辑的形式存在
- 执行 trace 作为模型输出的一部分

这是 `Parametric Execution` 的一种实现路线。

### 7.2 `WASM-as-Sandbox`

这表示：

- Wasm 模块作为外部执行对象存在
- 模块在 `WasmDomain` 中运行
- 访问系统资源需要 capability 授权
- 执行结果和 trace 由系统管理

这是 `Sandboxed Execution` 的一种实现路线。

### 7.3 两者不冲突

Kairo 不应在两者之间二选一。  
更合理的设计是：

- 用 `WASM-as-Weights` 提供模型内低延迟执行能力
- 用 `WASM-as-Sandbox` 提供可治理的系统级执行能力

## 8. 统一编排层

双层执行成立的关键，不只是同时拥有两条执行路径，而是存在一个统一编排层来做选择和连接。

### 8.1 编排层职责

编排层至少需要回答：

- 当前任务该走哪条执行路径
- 是否需要先做内部快速推导，再做外部受控执行
- 哪些结果需要写回上下文
- 哪些 trace 需要持久化
- 哪些动作需要权限批准

### 8.2 推荐对象

可以考虑引入以下对象：

- `ExecutionRequest`
- `ExecutionPolicy`
- `ExecutionLease`
- `ExecutionTrace`
- `ExecutionResult`
- `CapabilityGuard`

### 8.3 编排原则

编排层不应简单依据“是否能写成代码”来选择路径，而应依据：

- 是否需要外部副作用
- 是否需要审计
- 是否需要低延迟
- 是否需要可微或端到端训练
- 是否需要可独立升级

## 9. 如何选择执行路径

Kairo 可以把路径选择建模为一个策略问题。

### 9.1 优先走 Parametric 的情况

- 纯内部计算
- 高频小步骤
- 对延迟敏感
- 不涉及文件、网络、设备
- 不要求强审计

### 9.2 优先走 Sandboxed 的情况

- 访问外部资源
- 涉及副作用
- 需要权限管理
- 需要日志、审计、回放
- 需要独立部署模块

### 9.3 混合路径

很多任务最合理的做法是混合：

1. 模型内部做快速计算或计划生成
2. 生成 Wasm 模块或选择现有 Wasm 工具
3. 在 `WasmDomain` 中执行
4. 把结果和 trace 返回给 agent

这才是双层执行模型真正想支持的核心流程。

## 10. 与其他系统层的关系

### 10.1 与 Model Runtime Substrate 的关系

`Parametric Execution` 更贴近模型底座，因为它发生在模型内部执行过程中。  
它会使用：

- `InferenceSession`
- `ExecutionPlan`
- `TensorBuffer`
- `KvCache`

### 10.2 与 Execution Domains 的关系

`Sandboxed Execution` 更贴近执行域，特别是 `WasmDomain`。  
它会使用：

- 独立模块实例
- capability 边界
- 外部系统服务接口
- 审计 trace

### 10.3 与 Agent Runtime 的关系

Agent runtime 不应自己发明执行模型，而应建立在双层执行能力之上。  
它负责：

- 决策
- 编排
- 上下文绑定
- 权限申请
- 结果消费

## 11. 安全与治理

双层执行模型的最大风险，是把内部执行和外部执行边界搞混。

### 11.1 原则

- Parametric Execution 不应直接获得系统副作用权限
- Sandboxed Execution 必须通过 capability 获取外部资源访问权
- 内部 trace 和外部 trace 应分别建模
- 需要副作用的动作不能仅靠模型内部“假想执行”完成

### 11.2 审计要求

建议明确区分：

- `InternalExecutionTrace`
- `SandboxExecutionTrace`
- `AgentDecisionTrace`

这样后续才能分析：

- 模型内部想做什么
- 外部实际执行了什么
- agent 是如何做出路径选择的

## 12. 推荐源码骨架方向

未来代码结构可以逐步靠近：

```text
kairo-kernel/src/
├── substrate/
│   ├── model/
│   └── parametric/
├── domain/
│   ├── wasm/
│   ├── inference/
│   └── agent/
├── runtime/
│   └── orchestration/
```

语义上可理解为：

- `substrate/parametric/`：模型内执行能力
- `domain/wasm/`：外部 Wasm 沙箱执行
- `runtime/orchestration/`：统一编排与路径选择

## 13. 当前阶段建议

对 Kairo 当前阶段而言，双层执行模型最重要的作用，不是立刻实现两条完整执行路径，而是尽早防止系统架构退化为单一路径假设。

最值得先定下来的判断包括：

1. 不是所有程序执行都应该进入权重
2. 不是所有可计算任务都值得走外部沙箱
3. Wasm 在系统中可以同时作为“模型内 IR”和“外部沙箱格式”
4. agent runtime 应是双层执行的编排者，而不是唯一执行者
5. 系统需要统一的执行策略对象，而不是零散工具调用

## 14. 结论

Kairo 的双层执行模型，不是为了同时支持两种“实现技巧”，而是为了给未来 agent 系统提供两种本质不同但同样必要的能力：

- 模型内部的低延迟参数化执行
- 系统边界外的可治理沙箱执行

前者提供速度、融合度和可训练性；  
后者提供隔离、审计、权限控制和外部环境接入。

Kairo 的目标不应是在两者之间做取舍，而应提供统一的系统架构，让这两类执行在同一个 agent-native platform 中协同工作。
